TWO SURGICAL FIXES (each unblocked a major capstone path)
----------------------------------------------------------------------
The first smoke run surfaced TWO precise gaps; both got a small surgical fix
(the arc's "small fix that unblocks a major path is OK" allowance), each with
a unit test, neither touching the engine apply path or the Op wire encoding.

FIX 1 — SQL-standard doubled-quote string escape `''` (`kessel-sql` lexer).
  The seed FAILED (`cascade_seed`) the instant any value contained an
  apostrophe: psycopg2 quotes `bob's first post` as the SQL literal
  `'bob''s first post'` (PG §4.1.2.1 — a literal single quote is escaped by
  DOUBLING it). The single-quote lexer stopped at the FIRST inner `'`, so it
  truncated the string to `bob` and then choked on the trailing `s first
  post'` → `expected ',' or ')'`. The fix mirrors the already-present `"`
  delimited-identifier escape: a doubled `''` stays in the string and appends
  one `'`; a lone `'` closes it. A string with NO embedded quote is
  byte-identical to the pre-arc token (every prior literal KAT still passes).
  THIS IS A CORRECTNESS BUG that would bite ANY app whose data contains an
  apostrophe — names, titles, prose. Unblocked cascade_seed (and Q1/Q2/Q5
  which read `bob's first post` back).

  Diagnosed via direct psql against a BIGSERIAL table:
    INSERT … VALUES ('bobs post', 2)        → id returned (ok)
    INSERT … VALUES ('bob''s first post',2) → ERROR: expected `,` or `)`  (pre-fix)

FIX 2 — ORDER-BY-over-a-projection render (`kessel-pg-gateway`).
  After FIX 1, the seed + Q1/Q2/Q3/Q5/Q6 passed but Q4 surfaced a SECOND
  gap: `SELECT posts.title FROM posts ORDER BY posts.title LIMIT 2` →
  `projected row decode failed: projected row width 128 != expected 64`.
  Root cause: in `kessel-sql`'s SELECT lowering, the `_ if sort.is_some()`
  match arm fires BEFORE the `Proj::Cols(...)` arm, so ANY projection +
  ORDER BY compiles to `Op::SelectSorted` — which emits the FULL record
  stream (`[u32 len][full rec]*`, width 128 incl. header+bitmap+all fields),
  DROPPING the projection at the engine layer. The gateway's narrow
  `emit_projected_rows` decoder (which expects only the projected fields'
  concatenated width, 64 for one VARCHAR(64)) then mismatched. Fix: the
  gateway now detects this shape (`kessel_sql::select_projection_is_sorted`)
  and, for a sorted projection, decodes each FULL record against the table
  schema and re-projects the requested columns by index
  (`emit_projected_from_full_records`). Bonus: this path uses the record's
  null bitmap, so a projected NULL renders as a real PG NULL — STRICTLY
  better than the narrow path (which has no null mask). A non-sorted
  projection keeps the byte-identical narrow path. Unblocked Q4_paginate.

  Isolated via direct psql (pre-fix):
    SELECT title FROM posts                       → ok (Op::SelectFields, narrow)
    SELECT title FROM posts LIMIT 2               → ok (Op::SelectFields, narrow)
    SELECT title FROM posts ORDER BY title        → ERROR width 128 != 64 (sorted)
    SELECT id,title FROM posts ORDER BY title     → ERROR width 128 != 72 (sorted)
