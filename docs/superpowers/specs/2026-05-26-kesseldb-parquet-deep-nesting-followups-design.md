## 1. Problem

SP145 V1's per-shape compositional design accepts cross-products where the OUTER shape is List/Map/struct and the INNER shape is primitive, struct, or (for some outer shapes) the OTHER kind of group. Three specific cross-products were NOT yet supported, each rejected with a `SP146 follow-up` message:

| Today's rejection (after SP145) | What SP146 lifts | Anchor in source |
|---|---|---|
| `List<List<group>> ... 3+ deep LIST nesting: SP146 follow-up (V1 supports List<List<primitive>> only)` | `List<List<List<T>>>` (max_rep_level=3) | `classify_list_of_group` arm `LogicalType::List` |
| `List<Map<...>>: SP146 follow-up (V1 supports List<List/struct>)` | `List<Map<K, V>>` | `classify_list_of_group` arm `LogicalType::Map` |
| `MAP<_, Map> ...: SP146 follow-up` | `Map<K1, Map<K2, V>>` | `classify_map_of_group` arm `LogicalType::Map` (value side) |

There are additional cosmetic SP146 names in the source — `MAP<_, List<group>>`, `MAP<_, struct<group>>`, `List<struct> ... field is REPEATED`, `List<struct<group>>` — which are deeper compositional shapes whose primary unblockers are the three above. The three lifted by SP146 are the strict prerequisites; once they work, the deeper cross-products either compose naturally (if the inner shape is already supported) or remain as explicit reject paths.

Common technical property: each of the 3 shapes adds ONE more REPEATED ancestor to a leaf that the current SP145 per-shape assemblers don't yet model. Three new ColumnKind variants + three new assemblers + small classify-side recursive dispatch additions close them.

## 2. Goals and non-goals

**Goals (V1).**

- Lift the 3 named SP146 rejections in `classify_column_plan` (via `classify_list_of_group` + `classify_map_of_group`):
  - `List<List<List<T>>>` — extend `classify_list_of_group`'s `LogicalType::List` arm to accept inner-element groups that are themselves nested LISTs (recursion through `decode_field_by_kind`).
  - `List<Map<K, V>>` — replace the SP146-reject in `classify_list_of_group`'s `LogicalType::Map` arm with a new `NestedListOfMap` plan.
  - `Map<K1, Map<K2, V>>` — replace the SP146-reject in `classify_map_of_group`'s `LogicalType::Map` arm with a new `NestedMapOfMap` plan.
- Add 3 new assemblers in `assembly.rs`:
  - `assemble_list_of_list_of_list_primitive` (3-deep, max_rep_level=3).
  - `assemble_list_of_map_kv` (outer List + inner Map<K, V> with primitive leaves).
  - `assemble_map_of_map_kv` (outer Map + inner Map<K2, V> with primitive leaves).
- Wire the 3 new variants through `extract_nested`'s per-kind dispatch and `decode_field_by_kind`'s recursive variant so the recursive struct-field path also accepts them.
- 3 new pyarrow fixtures (`list_of_list_of_list_i64.parquet`, `list_of_map_string_i64.parquet`, `map_string_map_string_i64.parquet`) — generated by `tests/fixtures/regen_deep_nesting_followups.py`.
- Pentest coverage (~6-8 rows) for the new code paths.
- SP145 internal record (its design spec doubles as the record) — the 3 SP146 follow-ups marked CLOSED, OBJ-2c-5 arc marked FULLY CLOSED with NO remaining follow-ups.

**Non-goals (named, deferred).**

- Parquet logical types beyond LIST and MAP (interval, JSON, BSON) — physical-layer concerns handled at extract time; SP146 doesn't touch them.
- 4+ deep nesting (e.g. `List<List<List<List<T>>>>`, max_rep_level=4). Achievable by the same recursion pattern but no pyarrow corpus exercising it; would be SP147+.
- Maps with non-primitive KEYS (`Map<struct, V>`, `Map<List, V>`) — pathological per Parquet convention.
- struct fields that are themselves SP146 shapes (`struct<List<List<List<T>>>>` etc.) — naturally compose via `decode_field_by_kind` recursion once T2-T4 wire the new variants through, but each requires its own variant arm in `decode_field_by_kind`'s match. We pick up the common cases; rare ones remain reject paths until a real fixture demands them.

## 3. Architecture

### 3.1 Strategy: extend each existing per-shape assembler

Per SP145's design spec §3.3, the implementation strategy for cross-products is: **"the inner shape's assembly happens on a SUBSET of the parallel streams — only the leaves under that inner sub-tree, with rep/def levels offset to be relative to the inner sub-tree's root."**

For SP146, we extend each existing per-shape assembler one more recursion layer:

| SP146 shape | Outer assembler | New inner case |
|---|---|---|
| `List<List<List<T>>>` (max_rep_level=3) | extension of `assemble_list_of_list_primitive` | the rep=3 case becomes "continue innermost list" instead of an error — generalize the per-position cursor logic to a 3-level stack of accumulators. |
| `List<Map<K, V>>` (max_rep_level=2 on K leaf, =2 on V leaf) | `assemble_list_of_map_kv` (new) | per outer-list-item slot, build a `PqValue::Map` by zipping the K and V columns at the right slice — same K-V zip discipline as `assemble_map_kv` but driven by the OUTER List's rep stream. |
| `Map<K1, Map<K2, V>>` (max_rep_level=2 on inner K + V leaves) | `assemble_map_of_map_kv` (new) | per outer-map-entry slot, the V slot is itself a `PqValue::Map` built from the inner K and inner V column slices. |

### 3.2 Per-shape detail

**`List<List<List<T>>>` (max_rep_level=3).** Three REPEATED ancestors on the path → rep ∈ {0,1,2,3}. We extend the 6-case classifier in `assemble_list_of_list_primitive` to 8 cases (adding `MiddleNull`, `MiddleEmpty` between Outer and Inner). The accumulator becomes a 3-level stack: outer (list of middle-lists) → middle (list of inner-lists) → inner (list of items). Rep handling:

- `rep == 0` — flush all 3 levels, start new outer record.
- `rep == 1` — flush middle + inner; start new middle list within outer.
- `rep == 2` — flush inner; start new inner list within middle.
- `rep == 3` — continue innermost list (append item).

**`List<Map<K, V>>` (max_rep_level=2 on K, =2 on V).** The outer LIST adds +1 rep; the MAP middle adds +1 rep → both K and V leaves see max_rep_level=2. K and V column chunks share the SAME rep stream (they're both inside the same key_value middle group). We drive assembly off the V column's (rep, def) stream and consume K values in parallel — one K per "key_value middle present" event.

Rep semantics:
- `rep == 0` — new outer record. Flush previous outer-list of inner-maps.
- `rep == 1` — new item in outer list (which is a new Map<K, V>). Flush previous inner map.
- `rep == 2` — continue current inner Map<K, V> (append a (K, V) entry).

**`Map<K1, Map<K2, V>>` (max_rep_level=2 on inner K + V).** The outer MAP adds +1 rep; the inner MAP middle adds +1 rep. Outer K's max_rep_level=1 (only outer MAP middle is REPEATED for outer K). Inner K and inner V both see max_rep_level=2. We drive off the inner V's (rep, def) stream and consume outer K + inner K in parallel — one outer K per outer-map-entry boundary (rep=0 or rep=1), one inner K per "inner key_value middle present" event.

Rep semantics:
- `rep == 0` — new outer record. Flush previous outer-map.
- `rep == 1` — new entry in outer MAP (= new outer K + start new inner MAP).
- `rep == 2` — continue current inner MAP (append (inner_K, V) entry).

## 5. Acceptance criteria

1. `assemble_list_of_list_of_list_primitive` accepts max_rep_level=3 streams; full 3-deep nesting works.
2. `assemble_list_of_map_kv` accepts max_rep_level=2 on K + V leaves; outer List of inner Maps works.
3. `assemble_map_of_map_kv` accepts max_rep_level=2 on inner K + V leaves; outer Map of inner Maps works.
4. `classify_column_plan` accepts the 3 shapes (the SP146-named rejection messages no longer fire); existing 4 SP145 shapes still classify identically.
5. 3 new pyarrow fixtures roundtrip GREEN (first try, if the per-shape composition holds; honest fallback per §6 below if a fixture exposes a pyarrow shape quirk that the V1 design can't model — that fixture is skipped with `#[ignore]` + named SP147 follow-up).
6. Pentest matrix (~6-8 new rows) green; well-behaved Err for adversarial inputs.
7. SP145 internal record updated: the 3 SP146 follow-ups marked CLOSED; **OBJ-2c-5 arc FULLY CLOSED with NO remaining follow-ups**.
8. Workspace baseline (1085 default / 1118 featured) + ~10-15 new tests; default tree-grep EMPTY; binary protocol UNCHANGED.

## 6. Honest estimate + fallback

The per-shape compositional design generalizes cleanly to ONE more recursion layer for each of the 3 shapes. The fundamental risk: pyarrow's Parquet output for `Map<K, Map<K2, V>>` could nest the inner Map's REPEATED key_value group in a non-canonical way that doesn't match the simple "inner V column has max_rep_level=2" model. If that materializes, the slice ships the 2 shapes that work and skips the third with `#[ignore]` + a documented SP147 follow-up rather than blocking the slice on one quirky fixture.

The 3-deep `List<List<List<T>>>` shape has the deepest level math (4-5 def levels worth of branching) — most likely to need iteration before the classifier matches pyarrow's exact bit-packing. Hand-built KATs (T2) precede the pyarrow fixture (T5) so the math is verified before the real-data confrontation.
