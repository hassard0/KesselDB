HEADLINE
========
A real SQLAlchemy 2.0 declarative-ORM workload does NOT yet compose
end-to-end against KesselDB. The PG-wire substrate it sits on IS lit and
verified (engine.connect + Extended Query probe PASS; VARCHAR(n) DDL,
INSERT, and `SELECT *`[+WHERE] all PASS via psql, see below), but the
ORM's own emitted SQL hits three independent gaps that block every
ORM-managed operation:

  G1. create_all pg_catalog existence probe uses `relkind = ANY (ARRAY[...])`
      -> parser rejects `[` (`unexpected char '['`). Blocks ALL ORM DDL.
  G2. ORM SELECT always qualifies columns: `SELECT t.id, t.name FROM t`
      -> parser `expected FROM` (qualified `table.col` projection unsupported).
      Also the PG-wire render path only emits `SELECT * FROM <table>` —
      an explicit projection list (even unqualified `SELECT id,name FROM t`)
      returns "V1 PG-wire only renders `SELECT * FROM <table>`".
  G3. ORM UPDATE/DELETE qualify the WHERE column: `WHERE t.id = $1`
      -> parser `expected ID` (qualified `table.col` in WHERE unsupported).

Each gap is larger than a single surgical fix; named as follow-up arcs in T3.
The ONE pre-named surgical fix that WAS applied (VARCHAR(n) DDL alias)
is verified below but does not move the create_all needle because the
catalog probe (G1) fails BEFORE any CREATE TABLE is emitted.

----------------------------------------------------------------------
SURGICAL FIX SHIPPED (T2/T3, commit 36119f3)
----------------------------------------------------------------------
kessel-sql `kind_of`: `VARCHAR(n)` DDL alias -> `FieldKind::Char(n)`
(mirrors the existing BIGINT/INTEGER/SMALLINT/BOOLEAN SP-PG-CAT-T8
aliases). Pre-named in the design's fix rule (unblocks a MAJOR path:
every SQLAlchemy/Django/Rails/Diesel string column renders VARCHAR(n)).
KAT `pg_varchar_alias_maps_to_char` PASS on vulcan. The gateway already
encoded the varchar OID 1043 on the read/cast side; this closes the
write/DDL side. Bare `VARCHAR` w/o `(n)` rejects with a precise reason.

----------------------------------------------------------------------
NAMED FOLLOW-UP ARCS (gaps NOT fixed inline — each larger than surgical)
----------------------------------------------------------------------
SP-PG-SQL-QUALIFIED-COLS  — accept `table.col` in projection lists AND in
                            WHERE/SET (strip the redundant single-table
                            qualifier; JOIN already parses `t.c`). Unblocks
                            G2b + G3 — i.e. EVERY ORM SELECT/UPDATE/DELETE.
SP-PG-SQL-PROJECTION-RENDER — make the PG-wire render path emit an explicit
                            projection list (`SELECT c1,c2 FROM t`), not
                            just `SELECT *`. Unblocks G2a. (Parser already
                            has the SP-Analytic-Plan-MULTI projection
                            parser; the gateway render side is the gap.)
SP-PG-SQL-ANY-ARRAY       — recognise `col = ANY (ARRAY[...])` (and the
                            inverse `ANY(ARRAY[...]) = col`). Unblocks G1 —
                            SQLAlchemy's create_all/inspector existence
                            probe + IN-list lowering both use it.
SP-PG-DDL-VARCHAR-UNBOUNDED — bare `VARCHAR`/`TEXT`/`CHARACTER VARYING`
                            (no `(n)`, multi-word) DDL types.
SP-PG-DDL-VARCHAR-NATIVE  — true variable-length storage (vs fixed CHAR
                            alias) if CHAR-pad vs VARCHAR-trim semantics
                            surface in ORM round-trips.
SP-PG-RETURNING / SP-PG-SERIAL — server-generated/autoincrement PKs +
                            INSERT...RETURNING (not hit here — our model
                            uses explicit ids — but the next ORM model
                            shape will need them).

BOTTOM LINE: the PG-wire SUBSTRATE the ORM rides on is solid (connect,
EXTQ probe, VARCHAR DDL, INSERT, SELECT*[+WHERE] all green). The DECLARATIVE
-ORM layer is blocked by 3 SQL-SHAPE gaps in the kessel-sql parser /
PG-wire render path — qualified columns, explicit projection rendering,
and ANY(ARRAY[...]). Closing SP-PG-SQL-QUALIFIED-COLS + -PROJECTION-RENDER
+ -ANY-ARRAY would take this from 2/8 to a full declarative-ORM CRUD pass.
