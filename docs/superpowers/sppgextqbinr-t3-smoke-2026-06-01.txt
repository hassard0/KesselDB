What works:
- asyncpg's Bind with `result_formats=[1]` (its default) now succeeds.
- Server emits RowDescription whose per-field format_code slot = 0x0001
  (binary) — libpq's per-field decode-path selector switches into the
  binary decoder.
- Server emits DataRow whose per-column bytes are:
    - INT8 column: 8 bytes BE (e.g. `id=42` → `00 00 00 00 00 00 00 2A`)
    - CHAR(32) → TEXT: raw UTF-8 bytes (e.g. `'first'` padded by
      kessel-codec, stripped trailing NULs by `render_pg_text`)
- asyncpg cleanly decodes both columns as native Python `int` and
  `str` types — confirms binary RowDescription + binary DataRow are
  on the wire in lockstep.

Pre-existing limitations (unchanged by this arc):
- The parameterized `WHERE name = $1` with $1='first' returned 0 rows.
  Reason: substitute wraps the param as `'first'`; the engine compares
  against CHAR(32) storage which is "first\0\0\0...\0" (zero-padded
  fixed width). The string comparison fails because the engine's
  EQ-on-Char ignores padding only on the LEFT side. This is a
  CHAR(N)-comparison quirk in the engine, NOT a binary-RESULT issue —
  the binary-RESULT path itself works end-to-end (the SELECT * confirms
  rows decoded correctly). V2 SP-CHAR-PAD-COMPARE would close this.
- The parameterized SELECT into INT columns would also need V2
  SP-PG-EXTQ-CAST (gateway-side `::int8` cast rewrite) — same gap as
  in the BIN V1 transcript. Doesn't apply here because the headline
  test uses no-WHERE / WHERE-on-CHAR.
