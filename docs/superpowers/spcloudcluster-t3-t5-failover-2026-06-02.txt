=== honest notes ===

  ▸ kesseldb-cluster-0 (the newly recreated pod) initially logs
    `view=0 is_primary=true` because its `Replica` bootstraps in
    view=0 from a brand-new instance, before it sees view-change
    messages from peers. The cluster's actual primary in this
    moment is kesseldb-cluster-1 (view=1). Within seconds the new
    pod catches up via the peer transport and becomes a follower
    in view=1. (kesseldb-cluster-0 may CrashLoop briefly on a kind
    runner under CPU pressure — the bootstrap DNS retry loop +
    StatefulSet's restart-policy absorb it.)

  ▸ The kessel CLI's `--addrs` retry budget is
    `(len(addrs) * 4).max(8) = 12` rotations with 20ms gaps
    between attempts plus the follower's 5s server-side
    apply_raw budget. Total worst-case wall time for a single
    SQL across view-change is ~5-15s. The kind smoke landed
    within ~3s because the followers had already converged
    before the post-kill INSERT was issued.

  ▸ Cross-node exactly-once on SQL writes is NOT guaranteed
    (see T3 commit message) — the `[0xFE] ++ sql` path is not
    session-framed because the cluster server's session-frame
    path is `Op`-only. For strict exactly-once across primary
    kill, use Op-level `ClusterClient::call(&Op)` (session-
    framed). The T5 acceptance bar is "the cluster keeps
    serving + the SUM is correct," which the SQL surface
    delivers via the server's internal client_table dedup +
    the eventual-commit semantics of the SUM read.
