**Cahill SSI dangerous-structure recap (read once before T2).** Cahill et al. (SIGMOD 2008) characterise the unique anomaly possible under SI but not under serializability: a **dangerous structure** is two consecutive rw-antidependency edges Tx_X →rw Tx_pivot →rw Tx_Y where Tx_X and Tx_Y are concurrent with Tx_pivot. An rw-antidependency edge Tx_A →rw Tx_B means Tx_A read a key that Tx_B *subsequently wrote* — i.e., Tx_A's snapshot is "stale" with respect to Tx_B's write. Plain SI lets both Tx commit (no write-write conflict on disjoint keys); the dangerous structure proves the schedule is non-serializable. Cahill's theorem: aborting any one Tx in the structure restores serializability. KesselDB picks **abort the committing Tx** (Decision 3) — minimum SM-side churn (no commit undo needed). The structure forms when the committing Tx, on commit, gains BOTH an incoming AND an outgoing rw-edge OR makes a pre-existing pending Tx into such a pivot.

**The SM apply seam (gated on non-empty read_set, per Decision 4 empty-read_set fast-path).** The SP112 `Op::CommitTx` arm currently runs (in order): (i) `snapshot_opnum > commit_opnum` abort, (ii) per-write-set-key `has_version_in_range(snapshot, commit-1)` ww-conflict check + abort, (iii) per-write-set-key `put_versioned` install, (iv) return `TxCommitted`. S2.4 inserts a NEW step between (ii) and (iii): **(ii-bis) SSI dangerous-structure check** — gated on `read_set.is_empty() == false`. If the SSI check aborts, return `OpResult::TxAborted { DangerousStructure { other_commit_opnum } }`. If the SSI check passes, AFTER step (iii)'s `put_versioned` calls succeed, **register the new committed Tx in `pending_txs`** with the `(snapshot_opnum, read_set, write_set keys, has_incoming_rw: false, has_outgoing_rw: false)` shape (subject to any flag toggles done in step (ii-bis)).

**Window truncation timing (Decision 5).** The truncation runs at the **top of the SSI inner branch**, BEFORE the rw-edge derivation, so the derivation sees only in-window records. `let lo = commit_opnum.saturating_sub(MAX_TX_AGE); let _evicted = pending_txs.split_off(&lo);` — note `split_off` returns the keys >= lo (which we KEEP), so we must use the *opposite* pattern: `pending_txs = pending_txs.split_off(&lo);` to retain only `commit_opnum >= lo`. **Verify in T2 via a KAT.**

```rust
// !read_set.is_empty() so SP112 SI semantics are byte-identical
// for empty-read_set commits (Decision 4 empty-read_set fast path).

    // than the SSI lookback horizon. Decision 5: MAX_TX_AGE = 4096.
    let lo = commit_opnum.saturating_sub(MAX_TX_AGE);
    // BTreeMap::split_off(&k) returns the map with keys >= k. We want
    // to KEEP keys >= lo, so:
    self.pending_txs = self.pending_txs.split_off(&lo);

    // Step B: rw-edge derivation. Iterate every concurrent committed
    // Tx (commit_opnum in (snapshot_opnum, commit_opnum-1]) and
    // intersect its read_set/write_set with the committing Tx's
    // write_set/read_set. Build a key-only view of this Tx's
    // write_set (for the read_set intersection) and the committing
    // read_set (already keys-only).

        write_set.iter().map(|(t, o, _v)| (*t, *o)).collect();
    // write_set is already sorted (per SP112 BTreeMap iteration);
    // confirm by debug_assert if cheap.

    let mut has_outgoing = false;
    let mut has_incoming = false;
    let mut other_commit_opnum = 0u64;

    // Iterate concurrent Tx: commit_opnum > snapshot_opnum AND
    // commit_opnum < this commit's commit_opnum.
    let lo_range = snapshot_opnum.saturating_add(1);
    let hi_range = commit_opnum; // exclusive upper bound
    let concurrent: Vec<(u64, PendingTxRecord)> = self.pending_txs
        .range(lo_range..hi_range)
        .map(|(k, v)| (*k, v.clone()))
        .collect();

        //     → this Tx →rw Tx_A (this Tx has an outgoing rw-edge)

            if let Some(rec_mut) = self.pending_txs.get_mut(a_commit) {
                rec_mut.has_incoming_rw = true;
            }

                rec_mut.has_outgoing_rw = true;
            }

    // and outgoing rw-edges are set. OR: any pre-existing pending Tx
```

```rust
/// SP113 / S2.4: O(n + m) intersection check on two sorted slices.
/// Returns TRUE iff the slices share at least one element.
/// Deterministic: no hashing, no allocator state.
fn sorted_vec_intersects(a: &[(u32, [u8; 16])], b: &[(u32, [u8; 16])]) -> bool {
    let mut i = 0usize;
    let mut j = 0usize;
    while i < a.len() && j < b.len() {
        match a[i].cmp(&b[j]) {
            std::cmp::Ordering::Equal => return true,
            std::cmp::Ordering::Less  => i += 1,
            std::cmp::Ordering::Greater => j += 1,
        }
    }
    false
}
```
