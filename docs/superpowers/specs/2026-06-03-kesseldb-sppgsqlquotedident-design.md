# SP-PG-SQL-QUOTED-IDENT — accept SQL-standard double-quoted (delimited) identifiers

Date: 2026-06-03
Status: implemented
Parent arc: SP-PG-ORM-DJANGO (closed DONE_WITH_CONCERNS — named this as the P0 follow-up)

## 1. Context

The Django ORM smoke (`docs/superpowers/sppgormdjango-smoke-2026-06-03.txt`)
proved the engine + CRUD data path is fully Django-ready: fed the
UNQUOTED equivalent SQL, every Django-shaped CRUD operation
(autoincrement `INSERT … RETURNING id`, qualified `SELECT t.col`,
by-PK `UPDATE` / `DELETE`) passes. Django connect also passes (the
`set_config` surgical fix). But Django **unconditionally double-quotes
every SQL identifier** (`"smokeapp_author"."id"`, `"name"`, …) and
this is NOT configurable. kessel-sql's lexer had no double-quoted
identifier token, so it rejected the FIRST identifier of EVERY Django
statement with `unexpected char '"'`. Result: Django stuck at 2/8.

This is the single highest-leverage unlock — one lexer feature takes
Django's DDL + INSERT + SELECT + by-PK UPDATE/DELETE from "rejected at
the first token" to "executes".

## 2. Scope

The kessel-sql lexer accepts `"ident"` as a SQL-standard **delimited
identifier** everywhere a bare identifier works: table name, column
name, qualified reference (`"t"."col"`), and in projection / WHERE /
SET / FROM / ORDER BY / GROUP BY / RETURNING / INSERT column list /
CREATE TABLE column defs. Because the lexer is the single tokenizer
all of those grammar sites read, one lexer arm covers them all.

Delimited-identifier rules implemented (PG §4.1.1 / SQL-92):
- Case-PRESERVING (no folding).
- May contain characters a bare identifier can't (spaces, etc.).
- A literal `"` inside the identifier is written as a doubled `""`.
- A zero-length identifier `""` is rejected (PG rejects it too).
- An unterminated `"…` is a clean lexer error (not a panic / not a
  silent swallow of the rest of the statement).

## 3. V1 semantics decision: quoted vs bare identity

In stock PostgreSQL, a BARE identifier folds to lowercase while a
QUOTED identifier preserves case, so `"Name"` ≠ `name`. **KesselDB does
NOT fold bare identifiers** — the catalog stores names case-sensitively
(`type_named` uses `t.name == name`, column resolution uses
`f.name == col`), and the ONLY case-insensitive step anywhere is
keyword recognition (`eq_ignore_ascii_case` in `kw()` and the scanner
guards).

Therefore the V1 rule is the simplest correct one: **a quoted
identifier lowers to the exact same `Tok::Ident(contents)` a bare
identifier of that (already-correct-case) name would produce.** Quoting
is transparent at the compiled-`Op` layer — `compile("… FROM users …")`
and `compile(r#"… FROM "users" …"#)` produce byte-identical `Op`s
(locked by `quoted_select_compiles_same_op_as_bare`).

This is exactly the round-trip Django needs: Django quotes the SAME
names in its DDL and its DML, so `CREATE TABLE "t" ("id" …)` stores the
catalog name `t`/`id`, and `INSERT INTO "t" ("id") …` /
`SELECT "id" FROM "t"` resolve to those same names. Bare and quoted
spellings of the same name interoperate freely (a client may mix them).

## 4. V1 out of scope (documented)

- **Quoted keyword-as-identifier** (`SELECT "from" FROM t` with a
  column literally named `from`). Because a quoted identifier lowers to
  the same `Tok::Ident` as a bare one, a delimited identifier that
  SPELLS a reserved keyword in a position where the grammar also
  accepts that keyword is not distinguishable from the keyword. No ORM
  emits this, and KesselDB rejects keyword-spelled bare names at CREATE
  time, so it does not arise in practice. Strict fix: follow-up
  `SP-PG-SQL-QUOTED-KEYWORD` (would need a distinct `Tok::QuotedIdent`
  variant threaded through ~59 identifier-position match arms — large
  blast radius, deferred until a real workload needs it).
- **Embedded special chars beyond the catalog's allowance.** A
  delimited identifier may LEX with a space (`"My Col"`), but whether
  the catalog accepts such a name at CREATE TABLE is governed by the
  catalog's existing name-validation rules — unchanged here.

## 5. Acceptance

Django ORM smoke advances from 2/8 toward full CRUD (target ≥6/8). The
residual gaps after quotes are accepted are the already-named
follow-ups, chiefly `SP-PG-DDL-IDENTITY` (Django's default
`BigAutoField` renders `… GENERATED BY DEFAULT AS IDENTITY`, not
`BIGSERIAL`) and `SP-PG-SQL-AGG-ALIAS-RENDER` (`.count()`/`.aggregate()`).

## 6. Weak-spots (6+)

1. **Case-folding semantics (bare vs quoted).** Resolved by §3:
   KesselDB doesn't fold bare names, so quoted==bare for the same
   spelling. (A future PG-strict mode that lowercased bare names would
   break this equivalence — out of scope.)
2. **`""` escape.** Handled in the lexer (`"a""b"` → `a"b`), locked by
   `quoted_ident_doubled_quote_escape`.
3. **Quoted keyword as identifier** (`"select"` column). Documented §4
   out-of-scope; lowers to the keyword token. Follow-up
   `SP-PG-SQL-QUOTED-KEYWORD`.
4. **Quoted qualified refs** (`"t"."col"`). Each side is independently
   quoted; the existing `col_ident` qualifier-strip handles the `.`
   between two `Tok::Ident`s. Locked by `quoted_qualified_ref_lexes`,
   `quoted_select_compiles_same_op_as_bare`, `quoted_returning_detected`.
5. **Catalog identifier storage** (does the catalog preserve case?).
   Yes — case-sensitive `==`. The quotes are delimiters, NOT part of
   the stored name; `quoted_full_crud_round_trip` asserts the catalog
   stored the unquoted `t`.
6. **Interaction with the gateway-side raw-SQL scanners** (cast
   stripper, literal-cast validator, param rewriter, multi-statement
   splitter, insertmanyvalues `find_kw`). These scan raw SQL TEXT
   before / around kessel-sql and must skip quoted-identifier regions
   exactly like they skip single-quoted string literals — otherwise a
   `'` or `::` INSIDE a quoted identifier (`"O'Brien"`, `"a::b"`)
   mis-pairs the scanner. Audit result:
   - `extq::substitute::substitute_inner` + `count_placeholders` —
     ALREADY skip `"…"` (pre-existing). ✓
   - `query::contains_multiple_statements` — ALREADY skips `"…"`. ✓
   - `cast_stripper::strip_pg_casts_tracked` +
     `find_literal_cast_mismatch` — DID NOT skip `"…"`. **Fixed** this
     arc (+ KATs). The validator also resets `just_closed_string` after
     a quoted ident so a `::` after a quoted column ref isn't classified
     as a TEXT-literal cast.
   - `insertmanyvalues::find_kw` — now skips `"…"` defensively (Django's
     INSERT has no inner SELECT so it bails before this matters, but a
     `"FROM"`/`"VALUES"` column shouldn't false-match a keyword). ✓
   - `dispatch::leading_keyword_is` / `command_complete_tag` — read the
     leading bare keyword via `split_whitespace().next()`; ORM SQL never
     quotes the leading verb, so unaffected. ✓
   - `query::is_effectively_empty` — any non-comment byte (incl. `"`)
     makes the query non-empty; correct. ✓

## 7. Determinism

The lexer change does not touch the engine apply path. Quoted and bare
identifiers lower to the SAME `Tok::Ident`, so they compile to the SAME
`Op`; determinism (same Op → same apply) is preserved. No new external
deps; `#![forbid(unsafe_code)]` honored; HTTP/WS/binary surfaces
byte-untouched.
