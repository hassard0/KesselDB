--------------------- the actual wire shape ---------------------

SQLAlchemy's batched flush did NOT emit a plain multi-row VALUES INSERT.
It emitted its insertmanyvalues form (captured on the wire via the simple
Query protocol, literals inlined by psycopg2):

  INSERT INTO widgets (name)
  SELECT p0::VARCHAR
  FROM (VALUES ('a', 0), ('b', 1), ('c', 2)) AS imp_sen(p0, sen_counter)
  ORDER BY sen_counter
  RETURNING widgets.id, widgets.id AS id__1

The gateway's `insertmanyvalues::rewrite_insertmanyvalues` desugars this —
BEFORE the literal-cast validator (which would reject the `p0::VARCHAR`
projection cast) — into the plain multi-row form the engine already
handles:

  INSERT INTO widgets (name) VALUES ('a'), ('b'), ('c')
  RETURNING widgets.id, widgets.id AS id__1

The multi-row INSERT compiles to one Op::Txn of 3 serial Creates; the SM
assigns ids 1/2/3 deterministically on the apply thread and surfaces them
as OpResult::CreatedMany { ids: [1,2,3] }; the gateway emits 3 DataRows +
CommandComplete "INSERT 0 3". SQLAlchemy threads each id back onto its
flushed object.

--------------------- raw psycopg2 RETURNING proof --------------------

  INSERT INTO widgets (name) VALUES ('p'),('q') RETURNING id  -> [(3,), (4,)]   (multi-row RETURNING id)
  INSERT INTO widgets (name) VALUES ('x'),('y') RETURNING *   -> [(1,'x'),(2,'y')] (RETURNING * = all columns, N rows)
