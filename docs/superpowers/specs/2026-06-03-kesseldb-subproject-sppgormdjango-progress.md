## Headline

How much of a Django ORM CRUD workload works end-to-end against KesselDB?

- **connect** — **PASS** (new, this arc's surgical fix). Django's full
  connection-init (encoding probes + `SELECT set_config('TimeZone', $1,
  false)` timezone set + search_path) now completes. Previously the
  FROM-less `set_config` SELECT errored `expected FROM` and killed every
  `connection.cursor()` before any ORM op ran.
- **schema create** (`schema_editor.create_model`) — **GAP**. Django
  emits `CREATE TABLE "smokeapp_author" ("id" bigint ... GENERATED BY
  DEFAULT AS IDENTITY, "name" varchar(32) ...)` → `unexpected char '"'`
  (quoted identifiers) + would also hit `GENERATED ... AS IDENTITY`.
- **insert + autoincrement** / **select** / **get** / **update** /
  **delete** — **GAP**. Every ORM statement double-quotes identifiers →
  `unexpected char '"'`.

**The boundary is a single SQL-lexer gap.** Fed the equivalent UNQUOTED,
`BIGSERIAL`-spelled SQL, every Django-shaped operation PASSES:
autoincrement `INSERT … RETURNING id` → `[(1,)]`, qualified `SELECT
t.id, t.name FROM t`, by-PK `UPDATE`/`DELETE WHERE id = $1`. So the
engine + CRUD path is Django-ready; the gap is purely the SQL text
shape, not the data path.

## Surgical fix shipped

`pg_catalog::synthesize::synthesize_helper_function` now intercepts
`select [pg_catalog.]set_config('<name>', <value>, <is_local>)` and
returns a single text row (column `set_config`) — accept-and-echo the
requested value (KesselDB has no mutable session GUCs in V1; Django
ignores the text, it only needs the statement to succeed). Parity entry
added to `extq::scalar_row_descriptions::recognize_scalar_select` so the
Extended-Query Describe step agrees on the `("set_config", TEXT)` shape.
Same low-risk pattern-table approach as the existing `select 1` /
`version()` / `current_setting` intercepts.

## Named follow-up arcs (gaps)

- **`SP-PG-SQL-QUOTED-IDENT`** (P0) — kessel-sql lexer: accept
  `"ident"` delimited identifiers everywhere (table / column /
  qualifier in DDL / DML / projection / WHERE / SET / ORDER BY /
  GROUP BY / RETURNING / INSERT column list). Unblocks ~all Django ORM.
- **`SP-PG-DDL-IDENTITY`** (P1) — `GENERATED [BY DEFAULT|ALWAYS] AS
  IDENTITY` as a BIGSERIAL-autoincrement alias. Django 6's default
  `BigAutoField` renders IDENTITY, not BIGSERIAL.
- **`SP-PG-SQL-AGG-ALIAS-RENDER`** (P2) — `COUNT(*) AS "__count"` parse
  + FROM-ful aggregate-projection PG-wire render. Django `.count()` /
  `.exists()` / `.aggregate()`.
- **`SP-PG-DJANGO-INTROSPECT`** (P3) — `connection.introspection`
  (`table_names` / `get_table_description` / `get_constraints`) for a
  real `manage.py migrate` / `inspectdb` workload.
- **`SP-PG-SAVEPOINT`** (P3) — nested `transaction.atomic()`
  SAVEPOINT / RELEASE SAVEPOINT. KesselDB's txn model is an atomic
  non-interactive write batch (USAGE §6) with no nested savepoints —
  needs a compatibility decision.

## Honest boundary

Django reaches KesselDB and connects; the underlying
autoincrement/RETURNING/by-PK-CRUD engine path is fully Django-ready
when fed unquoted SQL. But Django quotes every identifier and KesselDB's
lexer rejects `"`, so 0/5 genuine ORM CRUD operations execute. The
entire Django ORM unlock funnels through one P0 follow-up. This is
strong, clean signal: KesselDB's real-world ORM readiness is one
lexer feature away from a second major Python ORM.
