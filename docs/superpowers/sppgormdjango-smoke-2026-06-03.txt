HEADLINE
========
A real Django 6 ORM workload reaches KesselDB and the CONNECTION-INIT
handshake now completes (the surgical `set_config` fix below), but the
ORM CRUD surface is BLOCKED end-to-end behind ONE lexer gap: Django
double-quotes EVERY SQL identifier and KesselDB's SQL tokenizer rejects
`"` with `unexpected char '"'`.

----------------------------------------------------------------------
SURGICAL FIX SHIPPED THIS ARC — `set_config(...)` connection-init GUC
----------------------------------------------------------------------
BEFORE: Django's PostgreSQL backend issues, on EVERY connection-init
(`_configure_timezone`), the FROM-less scalar SELECT

    SELECT set_config('TimeZone', $1, false)      -- $1 = 'UTC' (psycopg3)
    SELECT set_config('TimeZone', 'UTC', false)   -- inlined-literal form

kessel-sql rejects a bare FROM-less SELECT with `expected FROM`, so
EVERY Django `connection.cursor()` raised before any ORM op ran — the
entire Django path was dead on connect.

FIX: `pg_catalog::synthesize::synthesize_helper_function` now intercepts
`select set_config('<name>', <value>, <is_local>)` (both the
`pg_catalog.`-qualified and bare forms), mirroring the existing
`current_setting('<name>')` intercept. KesselDB has no mutable session
GUCs in V1, so it is a no-op set: accept-and-echo the requested value
(the 2nd arg if a quoted literal; else the canned GUC value for the
named setting — Django ignores the returned text, it only needs the
statement to succeed). A parity entry was added to
`extq::scalar_row_descriptions::recognize_scalar_select` so the
Extended-Query Describe step agrees on the `("set_config", TEXT)` column
shape (else a parameterized form would answer NoData and orphan the
DataRow). This is the lowest-risk shape-match fix (same pattern table as
`select 1` / `version()` / `current_setting`), and it unblocks the
entire Django connection lifecycle.

AFTER: `STAGE connect: PASS` — Django's full conn-init (encoding probes,
timezone set_config, search_path, etc.) completes cleanly.

----------------------------------------------------------------------
THE BLOCKING BOUNDARY — quoted identifiers (named follow-up, NOT fixed)
----------------------------------------------------------------------
Django emits (captured verbatim via schema_editor collect_sql + the ORM
compiler):

  DDL:    CREATE TABLE "smokeapp_author" ("id" bigint NOT NULL PRIMARY
          KEY GENERATED BY DEFAULT AS IDENTITY, "name" varchar(32) NOT NULL);
  SELECT: SELECT "smokeapp_author"."id", "smokeapp_author"."name"
          FROM "smokeapp_author"
  INSERT: INSERT INTO "smokeapp_author" ("name") VALUES ($1)
          RETURNING "smokeapp_author"."id"
  UPDATE: UPDATE "smokeapp_author" SET "name" = $1
          WHERE "smokeapp_author"."id" = $2
  DELETE: DELETE FROM "smokeapp_author" WHERE "smokeapp_author"."id" IN ($1)

EVERY statement double-quotes identifiers. KesselDB's SQL tokenizer
(kessel-sql) has no double-quoted-identifier token, so it errors with
`unexpected char '"'` on the FIRST identifier of EVERY Django statement.
Unlike SQLAlchemy (which by default emits UNQUOTED identifiers, so the
SQLAlchemy ORM smoke is 7/7), Django ALWAYS quotes — this is not
configurable. Quoted-identifier support is a kessel-sql lexer/parser
change that ripples across the ENTIRE SQL surface (DDL, DML, queries,
projections, WHERE, JOIN); it is explicitly OUT of scope for an inline
surgical fix and is named as a follow-up arc below.

PROOF THE ENGINE + CRUD PATH IS READY (quotes stripped, BIGSERIAL
spelling) — every Django-shaped operation PASSES once the two parser
gaps are removed:

  PASS  CREATE TABLE smokeapp_author (id BIGSERIAL PRIMARY KEY, name VARCHAR(32) NOT NULL)
  PASS  INSERT INTO smokeapp_author (name) VALUES ($1) RETURNING id     -> [(1,)]
  PASS  SELECT smokeapp_author.id, smokeapp_author.name FROM smokeapp_author  -> [(1,'tolkien')]
  PASS  SELECT ... WHERE smokeapp_author.id = $1                        -> [(1,'tolkien')]
  PASS  UPDATE smokeapp_author SET name=$1 WHERE id=$2
  PASS  DELETE FROM smokeapp_author WHERE id=$1

So KesselDB's autoincrement / RETURNING / qualified-column / by-PK
WHERE engine semantics are ALL Django-ready; the boundary is purely the
SQL TEXT SHAPE (quoting + IDENTITY spelling + COUNT render), not the
data path.

----------------------------------------------------------------------
NAMED FOLLOW-UP ARCS (gaps, in priority order)
----------------------------------------------------------------------
SP-PG-SQL-QUOTED-IDENT  (P0 — unblocks ~all Django ORM)
    kessel-sql lexer: accept `"ident"` as a delimited identifier
    (case-preserving, may contain reserved words / mixed case). Applies
    everywhere an identifier is parsed: table name, column name, in
    projection / WHERE / SET / ORDER BY / GROUP BY / RETURNING / INSERT
    column list. This single arc would take Django's DDL + INSERT +
    SELECT + by-PK UPDATE/DELETE green (the engine path is already
    proven, above).

SP-PG-DDL-IDENTITY  (P1 — Django default PK DDL)
    CREATE TABLE accept `<col> bigint ... GENERATED BY DEFAULT AS
    IDENTITY` (and `GENERATED ALWAYS AS IDENTITY`) as an alias for the
    BIGSERIAL autoincrement PK KesselDB already implements. Django 6's
    default `BigAutoField` renders IDENTITY, NOT BIGSERIAL — so even
    with quoted idents, the create_model DDL still needs this spelling.

SP-PG-SQL-AGG-ALIAS-RENDER  (P2 — Django .count()/.aggregate())
    `SELECT COUNT(*) AS "__count" FROM t` — two sub-gaps:
    (a) parse `COUNT(*) AS <alias>` (currently `expected FROM` on the
        bare-alias form, `unexpected char '"'` on the quoted alias), and
    (b) PG-wire render of a FROM-ful aggregate projection (currently
        "V1 PG-wire only renders SELECT * or an explicit projection").
    Django's `.count()`, `.exists()`, and `.aggregate()` all need this.

SP-PG-DJANGO-INTROSPECT  (P3 — Django migrations / inspectdb)
    Django's `connection.introspection` (table_names, get_table_description,
    get_constraints) drives `migrate` + `inspectdb`. Not exercised by
    this CRUD smoke (we use schema_editor.create_model directly), but
    required for a real `manage.py migrate` workload. Likely additional
    pg_catalog / information_schema shapes beyond the GUI-tool set.

SP-PG-SAVEPOINT  (P3 — Django atomic() nesting)
    Django wraps writes in SAVEPOINT/RELEASE SAVEPOINT for nested
    `transaction.atomic()` blocks. Not reached in this smoke (autocommit
    single statements), but a real Django request lifecycle uses them.
    KesselDB's txn model is an atomic non-interactive write batch with
    no nested savepoints (USAGE §6) — needs a compatibility decision.
