Model — NO explicit id (the real-world autoincrement default):

    class Widget(Base):
        __tablename__ = "widgets"
        id = Column(BigInteger, primary_key=True, autoincrement=True)  # no id on insert
        name = Column(String(32))

The application never supplies `id`; the DB assigns it; SQLAlchemy reads it
back via `INSERT INTO widgets (name) VALUES (...) RETURNING id`.

(engine created with use_insertmanyvalues=False so SQLAlchemy emits the
classic per-row `INSERT ... VALUES (...) RETURNING id` shape — the dominant
real-world ORM insert; its batched insertmanyvalues SELECT-VALUES-ORDER BY
form is a named follow-up SP-PG-RETURNING-MULTIROW.)
