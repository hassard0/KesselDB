**Read-your-writes overlay discipline (Decision 3).** When `Tx::read(t, o)` is called, the implementation MUST consult `self.write_set` BEFORE calling `mvcc::get_at_snapshot`. If `(t, *o)` is in the write_set:
- `Some(v)` → return `SnapshotRead::Found(v.clone())`.
- `None` → return `SnapshotRead::Tombstoned`.

In BOTH cases, the key MUST still be inserted into `self.read_set` (S2.2 Decision 4 — the Tx observed the key; SSI consumes both buffered-read observations and snapshot-read observations). This is gated by KATs in T2 (`kat_read_your_writes_returns_buffered_value`, `kat_read_your_writes_buffered_tombstone`, `kat_read_set_records_buffered_reads`).

**Conflict-check half-open-interval discipline (Decision 5).** The conflict check uses `mvcc::has_version_in_range(store, type_id, &object_id, lo_excl=snapshot_opnum, hi_incl=commit_opnum-1)`. The hi bound is `commit_opnum - 1` (not `commit_opnum`) because the Tx's own writes will install at `commit_opnum`; they must not conflict with themselves. The lo bound is `snapshot_opnum` EXCLUSIVE (the snapshot version itself was visible to the Tx and is not a conflict; only LATER versions invalidate the snapshot). **Edge case: `commit_opnum == 0`.** Then `commit_opnum - 1` would wrap to `u64::MAX`. The implementation MUST handle this explicitly: a Tx committing at `commit_opnum=0` cannot conflict with anything (no prior versions could exist). Skip the conflict-check loop entirely when `commit_opnum == 0`. Gated by a T5 pentest (`pt_commit_opnum_zero_skips_conflict_check`).

**SnapshotOutOfRange discipline (Decision 4 honest disclosure).** If `snapshot_opnum > commit_opnum`, the commit MUST return `Err(TxError::SnapshotOutOfRange { snapshot, commit })` BEFORE running any conflict check or installing any writes. This is the conservative response to a malformed input (a snapshot from the future is impossible in a well-formed Tx; only a hostile or buggy caller could submit one). The cursor-stall semantics for "snapshot_opnum <= current_opnum but > my locally-applied opnum" are deferred to S2.6 per the design's Decision 4 honest disclosure; S2.3's SM apply path assumes the snapshot has been applied. Gated by T5 (`pt_snapshot_greater_than_commit_opnum_rejected`).
